# 實作 Roadmap

給 Claude Code 的分階段實作指引。建議嚴格按順序執行，**每一階段完成都要能獨立運作並通過測試**。

## 階段總覽

| 階段 | 目標 | 預計時長 | 產出 |
|------|------|---------|------|
| 1 | 基礎產生器（單一素材） | 2-3 天 | sop-trainer 雛形 |
| 2 | 多素材整合與內訓增強 | 3-4 天 | sop-trainer 完整版 |
| 3 | Word/PDF 輸出 | 2 天 | 含品牌風格的文件 |
| 4 | 更新基礎（修改清單） | 3 天 | sop-updater 雛形 |
| 5 | 多素材更新與匯流 | 3-4 天 | 四種素材完整支援 |
| 6 | 三層級 changelog | 2 天 | 完整變更報告 |
| 7 | 整合測試 | 2 天 | 端對端可用 |

## 階段 1：基礎產生器（單一素材）

**目標**：用一份訪談逐字稿，產出基本的 IR 與 Markdown SOP。

### 工作項目

1. **建立 sop-trainer skill**
   - 撰寫 `skills/sop-trainer/SKILL.md`（觸發條件、工作流程）
   - 建立目錄結構

2. **實作 IR 結構**
   - 使用 `schemas/ir-schema.yaml` 做為驗證基礎
   - 寫一個 Python 腳本驗證 IR 是否符合 schema

3. **訪談逐字稿抽取器**
   - 輸入：純文字逐字稿
   - 輸出：IR YAML
   - 重點：抽取步驟、抽取 troubleshooting（從「之前有人弄錯」這類句式）

4. **基礎 Markdown 輸出**
   - 從 IR 產出可讀的 Markdown 文件
   - 包含目錄、步驟、術語表、變更紀錄

### 完成標準

- [ ] 用一份範例逐字稿能跑通整個流程
- [ ] IR 符合 schema
- [ ] 產出的 Markdown 可讀、結構正確
- [ ] 每個步驟都有 source_refs 追溯到逐字稿位置

### 測試素材

`tests/sample-transcript-aws-ec2.txt`（請手動準備一份 1500-3000 字的訪談範例）

## 階段 2：多素材整合與內訓增強

**目標**：支援訪談逐字稿 + 既有文件 + 截圖，並加入內訓增強機制。

### 工作項目

1. **既有文件抽取器**
   - 支援 Word（用 docx skill）
   - 支援 PDF（用 pdf-reading skill）
   - 重點：分類內容（步驟 / 背景 / 可丟棄）

2. **截圖處理**
   - 讓 Claude 看圖產生描述
   - 建立 `assets/images.yaml` 對映表
   - 截圖跟步驟的配對機制

3. **內訓增強 subagent**
   - 小新（Newbie Simulator）：模擬新人讀 SOP，標記看不懂的地方
   - 小修（Enhancer）：根據小新回饋補強
   - 小審（Reviewer）：最終一致性檢查

4. **完整性檢查**
   - 每步驟是否有 verification？
   - 是否有 rationale？
   - 術語首次出現是否定義？
   - gaps_detected 機制

### 完成標準

- [ ] 多素材混合輸入能整合成單一 IR
- [ ] 小新 → 小修 循環運作正常
- [ ] gaps_detected 能正確標記資訊缺口
- [ ] 截圖與步驟對應正確

## 階段 3：Word/PDF 輸出

**目標**：產出符合內訓品牌風格的 Word/PDF 文件。

### 工作項目

1. **Word 模板設計**
   - 使用 docx skill
   - 封面頁（含 metadata）
   - 目錄、頁首頁尾
   - 步驟用「卡片式」排版（含截圖、驗證、warning）
   - 警告框、提示框、操作框三種色塊

2. **品牌風格**
   - 主色：深色 accent（深藍/靛紫）+ 淺色底
   - 內訓友善：高對比、清晰字級、充足留白
   - Code block 樣式

3. **PDF 輸出**
   - 透過 docx → PDF 轉換
   - 確保截圖、表格、目錄正確呈現

4. **變更紀錄章節**
   - 自動附上最近 3 次變更摘要

### 完成標準

- [ ] Word 文件可直接交付給內訓使用
- [ ] PDF 排版正確、可印刷
- [ ] 截圖標註（紅框）正確顯示
- [ ] 含完整目錄與頁碼

## 階段 4：更新基礎（修改清單）

**目標**：用修改清單更新既有 SOP，產出新版 IR 與基本 changelog。

### 工作項目

1. **建立 sop-updater skill**
   - 撰寫 `skills/sop-updater/SKILL.md`
   - 互動式 CLI 流程

2. **舊 SOP → IR 反向解析**
   - 如果是本系統產的：直接讀 storage 中的 IR
   - 如果是外部 Word/PDF：解析重建（標記不確定欄位）

3. **修改清單抽取器**
   - 支援 Word 表格、Markdown 條列、Word 追蹤修訂
   - 產出 change_intents

4. **機械合併引擎**
   - 套用 change_intents 到舊 IR
   - 處理 add / modify / remove
   - 簡單的衝突偵測（明顯矛盾）

5. **基礎 changelog**
   - 列出本次變更的步驟
   - 每項變更附上 diff

### 完成標準

- [ ] 修改清單能正確產出 change_intents
- [ ] 機械合併不會漏改、不會錯改
- [ ] 新 IR 通過 schema 驗證
- [ ] step_id 跨版本保持穩定

## 階段 5：多素材更新與匯流

**目標**：支援文字檔、PDF、截圖三種額外素材，並做匯流處理。

### 工作項目

1. **文字檔抽取器**
   - 類型判斷（會議紀錄 / 變更說明 / 訪談 / 對話）
   - 依類型不同處理策略
   - 變更 vs. 補充的區分

2. **PDF 抽取器**
   - 用 pdf-reading skill
   - 處理表格、嵌入圖片
   - 術語對映表整合

3. **截圖抽取器**
   - 描述 → 配對 → 差異 → 意圖 四步驟
   - 視覺差異描述
   - 自動偵測「截圖變了，文字也要跟著變」

4. **匯流引擎**
   - 多源去重與信心提升
   - 衝突偵測（素材間矛盾）
   - 完整性檢查（改一半問題）

5. **互動式審核 CLI**
   - 變更意圖逐項確認
   - 衝突解決選單
   - 完整性問題處理

### 完成標準

- [ ] 四種素材能各自正確處理
- [ ] 多源印證能正確提升信心
- [ ] 衝突一定會拋出來給人看
- [ ] 完整性問題能被偵測

## 階段 6：三層級 changelog

**目標**：產出三種角色適用的變更報告。

### 工作項目

1. **變更分類**
   - 從 change_intents 推導出三種視角的內容

2. **管理者層級報告**
   - 一頁式高階摘要
   - 影響範圍、是否需重訓

3. **訓練員層級報告**
   - 變更項目表格
   - 每項說明對訓練的影響

4. **工程師層級報告**
   - 完整 diff
   - source_refs 追溯

5. **整合到 Word 輸出**
   - 內訓 SOP 自動附最近 3 次變更摘要
   - changelog 獨立 Word 檔可單獨分發

### 完成標準

- [ ] 三層級內容明確區分
- [ ] Word 排版專業
- [ ] 從 change_intents 自動產生，無需手寫

## 階段 7：整合測試

**目標**：端對端跑通完整流程，調校品質。

### 工作項目

1. **準備真實測試案例**
   - 至少 3 份不同類型的 SOP（AWS 操作、n8n 流程、教育訓練）
   - 至少 2 個更新案例（不同素材組合）

2. **端對端測試**
   - 從零產生 → 更新 v1.1 → 更新 v1.2
   - 檢查 step_id 穩定性
   - 檢查版本可重生性

3. **品質調校**
   - 信心閾值微調
   - prompt 優化
   - 邊界案例處理

4. **使用文件**
   - 給最終使用者的操作指南
   - 常見問題

### 完成標準

- [ ] 三份測試 SOP 都能正確產生與更新
- [ ] 跨版本 diff 正確
- [ ] 文件對非工程師也清楚

## 通用實作原則

### 給 Claude Code 的提醒

1. **每階段獨立可運作**：不要前置依賴未來階段才有的功能
2. **Schema 是契約**：任何資料結構變動先改 schema 與測試案例
3. **失敗要明顯**：出錯時拋明確訊息，不要靜默忽略
4. **不要幻覺欄位**：IR 沒有的資訊就留 null，不要編造
5. **保留中間產物**：抽取階段的中間 YAML 都要存下來方便除錯
6. **語言一律 zh-TW**：所有面向使用者的文字（含錯誤訊息）都用繁體中文

### 測試先行

每個抽取器、每個合併規則都要有測試案例。建議用 pytest 或簡單的 YAML 比對。

### 版本控制

整個專案用 git 管理。建議每階段完成後打 tag。

### 不要做的事

- 不要把 IR schema 寫死在程式裡，從 yaml 讀
- 不要在 Python 裡寫長 prompt，prompt 都放 skill 的 markdown
- 不要在第一階段就追求 100% 自動化，留人工閘門
- 不要為了速度跳過 source_refs 追溯，這是更新功能的命脈
