# Claude Code 啟動 Prompt 範本

> 給 Jin 用的：把這份規格交給 Claude Code 時，可以用以下 prompt 起手。

## 推薦的啟動 Prompt（複製貼上即可）

```
我要做一個 SOP 內訓文件產生與更新系統。

完整規格已經寫好放在這個資料夾，請按以下順序進行：

1. 先讀 CLAUDE_CODE_HANDOFF.md（這份是給你的接手指引）
2. 然後依該指引讀完所有規格文件
3. 跟我確認你理解的範圍與實作環境
4. 確認後從 docs/implementation-roadmap.md 的階段 1 開始實作

重要原則：
- 階段性實作，不要一次寫完
- 每階段完成都要展示給我看，確認後再進下一階段
- 規格有矛盾或不明確的地方，先問我，不要自己決定
- 所有面向使用者的文字一律繁體中文
- 時間戳一律 GMT+8

開始吧。
```

## 替代的較短 Prompt

如果你想更精簡：

```
請讀 CLAUDE_CODE_HANDOFF.md 然後依其指示進行。
```

CLAUDE_CODE_HANDOFF.md 會引導 Claude Code 讀完所有文件並向你確認。

## 第一階段啟動後常用的後續 prompt

### 確認後開始實作
```
確認，開始實作階段 1。
```

### 階段完成後進下一階段
```
看起來不錯，進入階段 2。
```

### 規格問題回答
```
這個情境採 {方案 A}，理由是 {...}。請更新規格文件，然後繼續實作。
```

### 要求 Claude Code 跑測試
```
跑一下 tests/sample-materials/transcripts/interview-aws-ec2.txt 看結果。
```

### 要求展示中間產物
```
給我看一下 sop-trainer 處理那份逐字稿後產出的 IR YAML。
```

## 給 Jin 的小建議

### 第一次啟動時

1. 把整個 `sop-system/` 目錄放在你想放的地方（例如 `~/projects/sop-system/`）
2. 在該目錄打開 Claude Code
3. 用上面的啟動 prompt
4. **不要急**，讓 Claude Code 充分閱讀規格再動工

### 進行中要注意的事

- 看到 Claude Code 「跳過確認直接寫程式碼」要打斷它，要求按規格來
- 看到輸出與規格不符要直接指出
- 規格本身的問題（例如有矛盾）也歡迎修正規格再繼續

### 階段間的休息點

每完成一個階段是好的休息點，可以：
- 把產出 commit 到 git
- 自己跑跑看確認可運作
- 想想下階段是否要調整需求

### 真實素材建議

建議第一個真實案例選擇：
- **長度適中**（不要太短沒挑戰，不要太長除錯困難）
- **你最熟悉**的領域（容易判斷對錯）
- **有完整素材**（最好是已經做過 SOP 的案例，可比對）

**強推**：用你的「AI 懶人報 n8n workflow」當第一個案例。你有：
- 訪談素材（自己寫的開發筆記）
- 既有文件（n8n 節點文件）
- 截圖（n8n 介面）
- 修改清單（之前做的更新）

這份案例剛好可以跑通整個系統。

## 如果遇到問題

### Claude Code 偏離規格

```
等等，這跟 docs/architecture.md 中 {章節} 的設計不符。
請回去看那段，然後解釋為什麼你選擇現在這個做法。
```

### Claude Code 進度卡住

```
你卡在哪？是規格不清楚，還是技術問題？把問題列出來，我們一起想。
```

### 想暫停 / 換方向

```
先停一下。我想{調整某個方向}。
請你先記錄目前進度到 docs/progress.md，然後我們討論。
```

### 規格更新

如果在實作中發現規格要改：
```
更新規格：{說明改什麼、為什麼}
請：
1. 先更新對應的規格檔案
2. 把已實作的部分調整
3. 確認測試案例還符合
4. 然後繼續實作
```
